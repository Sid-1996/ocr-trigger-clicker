# 條件清單 → Step 系統合併計畫

> 狀態：**強烈推薦執行** ｜ 建立日期：2026-07-15
> 
> 核心目標：消除「條件清單」與「Step 系統」並存的雙軌架構，將條件清單的能力**完全吸收**進 Step 系統，使條件清單從獨立模式降級為 Step 的一個 `condition_list` 類型。

---

## 一、問題根源分析

### 1.1 當前架構問題

```
Rule
├── steps: list[Step]          ← 正規流程控制（detect / click / key / wait / template_match / ...）
└── condition_list: list[?]    ← 平行於 steps 的另一套流程（僅在 use_condition_list=True 時生效）
```

**雙軌制帶來的問題：**

| 問題 | 說明 |
|------|------|
| **風格不貼合** | 條件清單是「條件→動作」配對，Step 是「依序執行」。兩套邏輯的執行模型不同（配對匹配 vs 線性流動），使用者被迫學習兩種心智模型 |
| **存在意義模糊** | `condition_list` 做的事（匹配條件→執行動作→推進索引）完全可以用 Step 的 `detect` + `on_match` / `on_no_match` 跳轉來表達 |
| **使用場景不輕便** | 條件清單只能在「整個 Rule」層級開關，無法與其他 Step 混搭；一個 Rule 要嘛全是條件清單，要嘛全是 Step，無法組合使用 |
| **維護成本高** | `05_main_loop.py` 中需要兩套執行路徑：`_execute_steps()` 和 `_execute_condition_list()`，任何新功能都要兩邊同步 |
| **序列化混亂** | Rule JSON 中 `use_condition_list` / `condition_list` / `condition_list_advance_on_no_match` 三個欄位和 `steps` 互斥但並存，增加驗證複雜度 |

### 1.2 為什麼「併入 Step 系統」是唯一正確解

條件清單的核心能力：
- **多條件依序匹配** → Step 本身已有 `detect` 類型
- **匹配成功執行對應動作** → Step 已有 `click` / `key` / `wait` 等動作類型
- **匹配失敗可選擇推進或停留** → 可用 Step 跳轉條件實現
- **索引持久化** → 可作為 Step 狀態的一部分

**結論：條件清單只是 Step 的一個特化子集，沒有獨立存在的必要。**

## 二、目標架構

### 2.1 合併後的 Rule 模型

```python
@dataclass
class Rule:
    id: str
    name: str
    enabled: bool
    steps: list[Step]          # ← 唯一的流程控制
    background: bool = False
    # ↓ 以下三個欄位將被移除
    # use_condition_list: bool = False           # 移除
    # condition_list: list | None = None          # 移除
    # condition_list_advance_on_no_match: bool = False  # 移除
```

### 2.2 新增 Step 類型：`condition_list`

```python
@dataclass
class Step:
    type: str       # 新增可選值: "condition_list"
    params: dict    # 包含 conditions、advance_on_no_match 等
```

**`condition_list` Step 的 params 結構：**

```json
{
  "type": "condition_list",
  "params": {
    "conditions": [
      {
        "detect": { "text": "目標A", "match_mode": "exact", "roi": {...} },
        "action": { "type": "click", "x": 100, "y": 200 },
        "on_match": "next_step"
      },
      {
        "detect": { "text": "目標B", "match_mode": "fuzzy", "fuzzy_threshold": 0.8 },
        "action": { "type": "key", "key": "enter" },
        "on_match": "repeat"
      }
    ],
    "advance_on_no_match": false,
    "loop": true
  }
}
```

### 2.3 Step 與 condition_list 混搭範例

```json
{
  "steps": [
    { "type": "detect", "params": { "text": "遊戲開始" } },
    { "type": "click", "params": { "x": 500, "y": 300 } },
    { "type": "wait", "params": { "ms": 1000 } },
    {
      "type": "condition_list",
      "params": {
        "conditions": [
          { "detect": { "text": "勝利" }, "action": { "type": "click", "x": 600, "y": 400 } },
          { "detect": { "text": "失敗" }, "action": { "type": "key", "key": "esc" } },
          { "detect": { "text": "繼續" }, "action": { "type": "click", "x": 300, "y": 500 } }
        ],
        "advance_on_no_match": false,
        "loop": false
      }
    },
    { "type": "wait", "params": { "ms": 500 } }
  ]
}
```

這樣就實現了：先用 detect 等進入某畫面，再用 condition_list 處理多分支選擇，最後 wait —— **完全在同一個 Step 序列中**。

## 三、執行階段

### Phase 1：向後兼容的資料模型擴展（1-2 天）

**目標**：讓系統同時接受新舊兩種格式，零中斷升級。

#### 任務清單

- [ ] **1.1** 在 `core/rule_models.py` 中新增 `Condition` 和 `ConditionListParams` dataclass
  ```python
  @dataclass
  class Condition:
      detect: dict       # detect step 的 params
      action: dict       # action step 的 params（含 type）
      on_match: str = "next_step"  # "next_step" | "repeat" | "jump:N"
  
  @dataclass
  class ConditionListParams:
      conditions: list[Condition]
      advance_on_no_match: bool = False
      loop: bool = True
  ```

- [ ] **1.2** 在 `core/rule_serialization.py` 新增 `condition_list` Step 類型的序列化/反序列化支援

- [ ] **1.3** 修改 `_dict_to_rule()`：當讀入舊格式 Rule 時，自動將 `condition_list` 欄位遷移為 `steps` 中的一個 `condition_list` Step
  ```python
  def _migrate_condition_list_to_step(rule_dict: dict) -> dict:
      """將 Rule 層級的 condition_list 遷移為 Step"""
      if rule_dict.get("use_condition_list") and rule_dict.get("condition_list"):
          condition_step = {
              "type": "condition_list",
              "params": {
                  "conditions": rule_dict["condition_list"],
                  "advance_on_no_match": rule_dict.get("condition_list_advance_on_no_match", False),
                  "loop": True
              }
          }
          rule_dict["steps"].append(condition_step)
          # 清除舊欄位
          for key in ["use_condition_list", "condition_list", "condition_list_advance_on_no_match"]:
              rule_dict.pop(key, None)
      return rule_dict
  ```

- [ ] **1.4** 修改 `_rule_to_dict()`：輸出時不再寫入 `use_condition_list` / `condition_list` / `condition_list_advance_on_no_match` 三個欄位

- [ ] **1.5** 單元測試：確保舊格式 JSON 載入後轉換為新格式，功能等價

### Phase 2：執行引擎重構（2-3 天）

**目標**：在 `05_main_loop.py` 中實現 `_execute_condition_list_step()`，刪除獨立的 `_execute_condition_list()` 路徑。

#### 任務清單

- [ ] **2.1** 在 `05_main_loop.py` 中新增 `_execute_condition_list_step(rule, step, state)` 函數

  - 支援 `on_match` 跳轉（`next_step` / `repeat` / `jump:N`）
  - 支援 `loop` 參數（是否循環檢查條件）

- [ ] **2.2** 修改 `_execute_steps()` 主循環，在 Step type 為 `condition_list` 時呼叫新函數

- [ ] **2.3** 刪除 `_execute_condition_list()` 獨立路徑（約 50-80 行程式碼）

- [ ] **2.4** 刪除 `Rule.use_condition_list` 判斷分支（`05_main_loop.py` 中 `if rule.use_condition_list:` 的所有區塊）

- [ ] **2.5** 整合測試：

## 四、風險評估與緩解

| 風險 | 機率 | 影響 | 緩解措施 |
|------|------|------|----------|
| 舊任務檔遷移失敗 | 低 | 高 | Phase 1 自動遷移 + 遷移前自動備份原檔 |
| condition_list Step 效能不如原生實現 | 中 | 中 | 在 `_execute_condition_list_step()` 中保留原有的索引快取機制 |
| GUI 改動過大影響使用者習慣 | 中 | 中 | 保持 condition_list Step 的 UI 與原「條件清單」編輯器視覺相似 |
| 迴歸錯誤（舊 Rule 行為改變） | 中 | 高 | Phase 2 完成後用現有任務庫做完整迴歸測試 |

## 五、驗收標準

1. **功能等價**：所有現有使用 condition_list 的任務在新架構下行為完全一致
2. **混搭可用**：能在一個 Rule 的 steps 中任意穿插 `detect`、`click`、`condition_list`、`wait`
3. **無獨立路徑**：`05_main_loop.py` 中不再有 `if rule.use_condition_list:` 分支
4. **模型簡化**：`Rule` dataclass 不再包含 condition_list 相關欄位
5. **向後兼容**：舊 JSON 格式的任務檔可無損載入

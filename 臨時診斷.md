# 專案全面診斷報告

---

## 一、專案概述

本專案為 **OCR Trigger Clicker** — 一個基於 OCR 辨識的自動化點擊工具，主要功能為：
- 擷取指定視窗畫面
- 使用 OCR 引擎辨識文字
- 根據使用者定義的規則（目標文字、區域、點擊位置等）執行自動點擊
- 提供圖形化介面（PyQt6）進行規則編輯、診斷與監控

---

```
  ├── 01_screenshot.py        # 螢幕/視窗截圖、座標轉換

```

## 三、詳細診斷（按模組）

### 1. `06_gui_main.py` – 主視窗

**優點**
- 使用 `QThread` 執行初始化，避免 UI 凍結
- 透過信號（`WorkerSignals`）跨線程更新 UI，設計正確
- 支援多任務管理（新增、刪除、匯入／匯出）
- 提供快捷鍵（F9 暫停/繼續，F12 緊急停止）
- 使用 `_NoWheelCombo` / `_NoWheelSpin` 防止滑鼠滾輪誤觸

**潛在問題與建議**

| 問題 | 風險等級 | 建議 |
|------|----------|------|
| `_restore_last_state` 中，若上次視窗已消失，僅顯示提示但未清空 combo | 低 | 可在找不到時清空 `_window_combo` 並設 placeholder |
| `_on_task_save` 儲存後呼叫 `reload_rules()`，但若主循環未啟動則無作用，可能造成規則不同步 | 中 | 應統一透過 `_load_task` 重載，或檢查 `_loop` 是否非空 |
| `_add_rule` 與 `_delete_rule` 限制「正在執行時不可操作」，但未考慮「暫停」狀態 | 低 | 暫停時也應禁止編輯，或提示“請先停止” |
| `_save_current_rule` 未驗證 `click_position == 'custom'` 時是否有有效座標 | 中 | 若為 custom 且座標為 (0,0)，應警告使用者 |
| `_emergency_stop` 中呼叫 `_ahk_mod.shutdown()` 內含 `os._exit(0)`，會強制終止整個 Python 程序 | **嚴重** | 切勿在 GUI 應用中呼叫 `os._exit`，應僅關閉 AHK 子程序並重置狀態 |
| `closeEvent` 中 `_ahk_mod.shutdown()` 同樣會終止程序，應改為優雅關閉 socket 與子進程 | 嚴重 | 修改 `shutdown` 為非強制退出 |
| 未處理多螢幕 DPI 縮放對 `QApplication` 的影響 | 中 | 應設定 `Qt.AA_EnableHighDpiScaling` 等屬性 |
| 狀態列顯示 CPU/MEM 時，`_perf_label` 樣式變更後未恢復 | 低 | 可在數值回歸正常時重置顏色 |
| `_init_worker` 在 `_on_init_finished` 成功後未清理（可能造成記憶體洩漏） | 低 | 可設為 `None` |
| `_debug_panel` 與主視窗生命週期綁定，但未在關閉時清理 | 低 | 可忽略，但建議在 `closeEvent` 中調用 `_debug_panel.deleteLater()` |

### 6. `01_screenshot.py` – 截圖模組

**優點**
- 支援多種後備方案（mss → PrintWindow → BitBlt）
- 處理 DPI 縮放
- 提供 `get_window_client_offset` 計算客戶區偏移

**潛在問題**

| 問題 | 風險 | 建議 |
|------|------|------|
| `get_dpi_scaling_factor` 使用 `SetProcessDpiAwareness(1)` 可能影響其他模組 | 中 | 應在應用啟動時統一設定，避免重複呼叫 |
| `capture` 中若啟用 `roi`，會將 ROI 乘以 DPI 縮放，但若 ROI 是視窗相對，此處乘縮放正確；但主循環中 `_process_rules` 傳入的 ROI 未乘縮放，因此這裡再做一次乘法可能導致偏移 | **關鍵** | 檢查 `_process_rules` 中呼叫 `recognize(roi_img, ...)` 時，roi 已經是視窗相對像素（未乘縮放），但 `capture` 內部若傳入 ROI 時會乘縮放，這與 `capture` 的設計一致。然而，主循環中從未使用 `capture` 的 ROI 參數，而是直接截取全視窗後再裁切 ROI 影像，因此 `capture` 的 ROI 參數實際上未被使用。但此處設計易混淆，建議移除或明確說明。 |

| `_pw_render` 使用 `PrintWindow` 可能對某些 DirectX 視窗無效 | 低 | 可保留現有 fallback 機制 |

### 7. `02_ocr_engine.py` – OCR 引擎

**優點**
- 支援多種後端（RapidOCR、EasyOCR、CnOCR）
- 使用 ThreadPoolExecutor 並行處理，有超時控制
- 健康檢查機制（連續失敗重啟引擎）
- Monkey-patch 解決 V5 模型寬度限制問題

**潛在問題**

| 問題 | 風險 | 建議 |
|------|------|------|
| `_run_engine` 中直接列印耗時，若頻繁 OCR 可能造成大量輸出 | 低 | 可改用 logging 或 debug 開關 |
| `_patched_resize` 僅修正 V5 模型的寬度限制，但若模型版本變更可能無效 | 中 | 應檢查模型版本或統一調整 `rec_img_shape` |
| `init_engine` 中 warmup 時若失敗，僅印出 traceback 但繼續執行，可能導致首次正式 OCR 失敗 | 中 | 應 catch 例外並重試或設置標記 |
| `recognize` 中對 `image` 進行 `cv2.resize` 時若影像為空會拋錯，但已檢查 `image.size` | 無 | 已有保護 |
| 使用 `ThreadPoolExecutor` 僅一個 worker，若多個執行緒同時呼叫 `recognize`，會佇列排隊，可能影響即時性 | 中 | 可考慮增加 worker 數量或使用非同步 Queue |
| 字典檔 `chinese_cht_dict.txt` 檔案較大（約 2MB），每次載入可能耗時 | 低 | 可考慮壓縮或僅載入需要的字符 |

### 8. `03_ahk_socket.py` – AHK 通訊

**優點**
- 支援重新連線與重啟 AHK 進程
- 心跳機制檢測連線健康
- 緊急停止指令 `ESTOP`
- 座標邊界檢查，防止點擊超出螢幕

**潛在問題與建議**

| 問題 | 風險 | 建議 |
|------|------|------|
| `_emergency_stop` 設定 `_heartbeat_event` 並關閉連線，但 `_restart_lock` 可能造成死鎖 | 中 | 使用 `try-finally` 確保鎖釋放 |
| `_restart_ahk` 中嘗試獲取鎖時若失敗，返回 False，但呼叫端未處理此情況 | 中 | 在 `_heartbeat_loop` 中若 `_restart_ahk` 返回 False，應增加失敗計數並最終呼叫 `_emergency_stop` |
| `shutdown` 最後呼叫 `os._exit(0)`，會強制結束 Python 程序，非常危險 | **嚴重** | 應只關閉 socket 和 kill AHK 進程，不應退出整個進程。GUI 關閉時，只調用 `_close_all` 和終止心跳執行緒即可 |
| `_find_ahk_executable` 從註冊表查找，但若使用者安裝為 Portable 版本則找不到 | 中 | 可允許使用者手動指定 AHK 路徑 |
| `_launch_ahk` 建立子程序時未設置 `stdin/stdout/stderr`，可能導致 AHK 錯誤訊息遺失 | 低 | 可導向 `subprocess.PIPE` 並記錄 |
| `_recv_line` 中若連線中斷，會跳出迴圈返回空字串，但未偵測斷線 | 中 | 應檢查返回值長度，若為空則視為斷線 |

### 10. `05_main_loop.py` – 主循環

**優點**
- 具有效能監控與速率限制
- 支援 `focus_safe` 僅在前景點擊
- 畫面變動偵測（frame diff）跳過無變化畫面
- 緊急停止與暫停機制
- 規則失控保護（runaway detection）

**潛在問題與建議**

| 問題 | 風險 | 建議 |
|------|------|------|
| `_process_rules` 中呼叫 `recognize(roi_img, ...)` 時未傳遞 `roi_offset`，導致回傳的座標是相對於 ROI 影像，之後直接加 `x1, y1` 修正，這是正確的。但若 ROI 為 None（全視窗），則傳入 `recognize` 為全圖，座標為全圖座標，也正確。 |
| `_process_rules` 中對每個規則重複執行 OCR（全圖或 ROI），若有多個規則共用相同 ROI 則浪費 | 低 | 可快取同一 ROI 的 OCR 結果 |
| `capture` 失敗時 fallback 到 `capture_window_content`，但後者回傳的影像可能不包含邊框，雖有填補邏輯，但若偏移計算有誤可能導致座標錯位 | 中 | 加強測試 `get_window_client_offset` 的準確性 |
| `_loop` 中每次迭代先 `get_window_rect`，若視窗消失則暫停，但暫停後不會自動恢復（需手動重啟） | 中 | 可提供「自動重試」選項，定時嘗試重新獲取視窗 |
| `_process_rules` 中檢查 `rule.enabled` 和 `rule.id in self._rule_auto_disabled`，但 `_rule_auto_disabled` 在規則重新載入時未清空 | 中 | 在 `reload_rules` 時應清空 `_rule_auto_disabled` |
| `_send_click` 直接呼叫 `_ahk.send_click`，若 AHK 離線則靜默失敗 | 中 | 應記錄錯誤或嘗試重啟 AHK |
| `_frame_diff_ratio` 設定為 0.02，若畫面文字變動小但有意義（如數字增加），可能跳過 OCR | 低 | 可考慮更靈敏的變化檢測（如文字區域）或移除跳過機制 |
| 主循環未處理規則儲存後的同步問題（如 `apply_trigger` 修改 `rule.trigger_count` 但未儲存） | 中 | 應在每次觸發後儲存規則檔案（或定期儲存） |

## 四、綜合問題與建議

### 架構層面
1. **緊急停止時強制退出程序**：`ahk_socket.shutdown()` 中的 `os._exit(0)` 須立即移除，改為優雅關閉所有資源。
2. **多螢幕支援不足**：ROI 和點擊座標選取器未正確處理多顯示器，建議改用虛擬桌面座標（虛擬螢幕邊界為負值）。
3. **規則狀態同步**：`MainLoop` 修改規則後（如觸發次數、啟用狀態）未寫回檔案，導致重啟後狀態遺失。應在每次觸發後儲存，或提供「儲存狀態」按鈕。
4. **錯誤恢復機制薄弱**：OCR 引擎崩潰、AHK 連線中斷時雖有重啟，但 GUI 未充分反映，使用者難以察覺。

### 性能與資源
1. **記憶體使用**：`_latest_raw` 保留高解析度截圖，OCR Debug 面板可能累積多張影像，建議限制快取張數。
2. **GDI 資源泄漏**：截圖模組中的 `GetDC` / `CreateCompatibleDC` 已正確釋放，但 `capture` 中 mss 可能未釋放資源（`with` 確保）。
3. **CPU 使用**：主循環即使畫面無變化仍持續執行 `get_window_rect` 和 `capture`，可考慮降低偵測頻率。

### 使用者體驗
1. **初始化時間長**：OCR 引擎首次載入模型耗時，應顯示進度條。
2. **錯誤訊息不明確**：許多地方僅印出例外，未顯示給使用者（如 OCR 失敗）。
3. **任務切換**：切換任務時規則列表刷新，但若正在執行中，應提示使用者停止後再切換。
4. **規則編輯**：點擊座標若為 custom 但未設置，儲存時無警告。

### 安全性
1. **AHK 腳本路徑**：寫死為 `clicker.ahk`，若執行目錄不可寫可能失敗。
2. **伺服器連接埠**：固定為 12345，可能與其他程式衝突，可改為動態分配。

## 六、測試建議

1. **多螢幕環境**：驗證 ROI 與點擊座標選取是否正確對應到副螢幕視窗。
2. **高 DPI 縮放**：測試 125%、150%、200% 縮放下截圖與點擊準確度。
3. **AHK 斷線恢復**：手動終止 AHK 進程，觀察自動重啟與連線恢復。
4. **長時間運行**：檢查記憶體洩漏（使用 Windows Performance Monitor 追蹤）。
5. **規則過載**：建立多條規則高頻觸發，測試速率限制與失控保護。

## 七、結語

本專案核心邏輯設計穩固，模組劃分清晰，具備良好的錯誤處理和性能監控。主要風險集中於**進程管理**（強制退出）和**多螢幕相容性**，建議優先修正。其餘問題多為邊界情況或 UX 優化，可逐步改善。

# 修復計畫

## 修復順序

```
Phase 1: P0 緊急 (1 項)  ─→  Phase 2: P1 高優先 (3 項)  ─→  Phase 3: P2 中優先 (6 項)  ─→  Phase 4: P3 低優先 (1 項)
```

每項修復獨立，原則上無前後依賴關係，可按任意順序執行。

---

## Phase 1 — P0 緊急

### F1: 移除 `shutdown()` 中的 `os._exit(0)`

| 項目 | 內容 |
|------|------|
| **檔案** | `core/03_ahk_socket.py` |
| **行數** | 310–324 |
| **問題** | `shutdown()` 最後呼叫 `os._exit(0)`，會不經任何清理直接終止 Python 行程。由 `closeEvent` (`gui/06_gui_main.py:1019`) 觸發時，可能會中斷正在寫入的檔案或釋放中的資源 |
| **修複** | 移除 `import os as _os` 與 `_os._exit(0)`。`_close_all()` 已負責關閉 socket 與 kill AHK 子行程，無需強制退出 |
| **驗證** | 關閉主視窗後行程應正常結束；手動確認 `_close_all()` 有被呼叫到 |

## Phase 2 — P1 高優先

### F2: 多螢幕 + DPI 座標一致性修正

| 項目 | 內容 |
|------|------|
| **檔案** | `gui/07_gui_roi.py`、`gui/13_gui_click_picker.py`、`gui/06_gui_main.py` |
| **問題** | 兩個子問題需一併處理：<br>① **多螢幕覆蓋不全**：overlay 只設置在 `primaryScreen().virtualGeometry()`，副螢幕區域無法互動<br>② **DPI 座標系不一致**：Qt6 overlay 回傳邏輯像素，但 `get_window_rect` (底層 `GetWindowRect`) 回傳實體像素。非 100% 縮放時座標轉換會偏移 |
| **修複內容** | |

#### F2a — Overlay 幾何覆蓋所有螢幕

**`07_gui_roi.py:19–35`** 與 **`13_gui_click_picker.py:16`**

現狀：
```python
geometry = QApplication.primaryScreen().virtualGeometry()
self.setGeometry(geometry)
screen = QApplication.primaryScreen()
self._bg_pixmap = screen.grabWindow(0, geometry.x(), geometry.y(), geometry.width(), geometry.height())
```

改為（ROI 選擇器為例，點擊選取器比照辦理）：
```python
from PyQt6.QtCore import QRect

# 計算所有螢幕的聯合虛擬桌面範圍
all_geometry = QRect()
for screen in QApplication.screens():
    all_geometry = all_geometry.united(screen.geometry())
self.setGeometry(all_geometry)

# 逐螢幕截圖並合成背景
self._bg_pixmaps = []
self._screen_geometries = []
for screen in QApplication.screens():
    geom = screen.geometry()
    pixmap = screen.grabWindow(0, 0, 0, geom.width(), geom.height())
    self._bg_pixmaps.append(pixmap)
    self._screen_geometries.append(geom)
```

`paintEvent` 改為逐一繪製各螢幕背景。

#### F2b — Overlay 座標轉實體像素後回傳

**`06_gui_main.py` 兩處轉換邏輯**：

① `_on_pick_coord` (行 795–799)：點擊選取器回傳 → 轉視窗相對
```python
# 新增：將邏輯像素轉為實體像素
screen = QApplication.primaryScreen()
ratio = screen.devicePixelRatio()
result = (int(result[0] * ratio), int(result[1] * ratio))

wr = get_window_rect(title)
if wr:
    result = (result[0] - wr["x"], result[1] - wr["y"])
```

② `_open_roi_selector` (行 819–825)：ROI 選擇器回傳 → 轉視窗相對（比照同上）

| 驗證 | 方法 |
|------|------|
| 雙螢幕環境 | 在副螢幕的視窗上框選 ROI，確認座標正確對應 |
| 125%/150%/200% 縮放 | 選取座標後啟動循環，觀察點擊位置是否準確 |

#### F3a — 允許持久化執行期欄位

**`04_rule_engine.py:98–102`**：不再從 `_rule_to_dict` 排除 `trigger_count` 與 `last_trigger_time`

```python
def _rule_to_dict(r: Rule) -> dict:

```

#### F3b — 觸發後寫回檔案

**`05_main_loop.py:197–209** (`_process_rules` 中 `apply_trigger` 之後)：
```python
params = apply_trigger(rule)
# ... 發送點擊 ...
ok = self._send_click(sx, sy, params["button"])
if ok:
    self._perf.record_click()

# === 新增：每次觸發後同步儲存規則 ===
with self._rules_lock:
    save_rules(self._rules, self._rules_path)
```

若擔心效能，可改為批次儲存（每 5 秒或每累積 N 次觸發才寫入一次）。

| 驗證 | 方法 |
|------|------|
| 規則 `trigger_mode=once` | 觸發後重啟程式，確認該規則仍為停用狀態 |
| 規則 `max_triggers=3` | 觸發 3 次後重啟，確認 `trigger_count` 保留為 3 |

## Phase 3 — P2 中優先

### F5: `reload_rules()` 時清空 `_rule_auto_disabled`

| 項目 | 內容 |
|------|------|
| **檔案** | `core/05_main_loop.py` |
| **行數** | 355–357 |
| **問題** | `reload_rules()` 僅重讀檔案，不清空 `_rule_auto_disabled` set。若某規則之前被失控保護停用，編輯規則後重載仍會被跳過 |
| **修複** | 在 `_load_rules` 或 `reload_rules` 中加入 `self._rule_auto_disabled.clear()` |
| **驗證** | 手動觸發失控保護停用規則 → 編輯規則內容 → 重新載入 → 規則應可重新觸發 |

### F6: DPI 縮放因子初始化統一化

| 項目 | 內容 |
|------|------|
| **檔案** | `core/01_screenshot.py` |
| **行數** | 122–145 |
| **問題** | `get_dpi_scaling_factor` 每次被呼叫都嘗試 `SetProcessDpiAwareness(1)`，可能與其他模組衝突。且 `capture()` 在每次截圖時都呼叫此函式 |

| **驗證** | 多 DPI 縮放下截圖尺寸與座標正確，無 `OSError` 拋出 |

### F7: 視窗消失後自動恢復偵測

| 項目 | 內容 |
|------|------|
| **檔案** | `core/05_main_loop.py` |
| **行數** | 247–254 |
| **問題** | 視窗被關閉時僅暫停並等待，不會自動恢復。使用者必須手動重啟 |
| **修複** | 在暫停迴圈中加入週期性重試（例如每 5 秒呼叫一次 `get_window_rect`），若視窗重新出現則自動恢復。注意需要限制重試頻率避免耗盡 CPU |
| **驗證** | 啟動偵測後關閉目標視窗 → 重新開啟目標視窗 → 自動恢復偵測 |

### F8: 儲存自訂座標前驗證有效性

| 項目 | 內容 |
|------|------|
| **檔案** | `gui/06_gui_main.py` |
| **行數** | 756–781 (`_save_current_rule`) |
| **問題** | `click_position == 'custom'` 但使用者從未點擊「選取點擊座標」時，`custom_x/custom_y` 為 (0, 0)，可能導致游標跳到左上角 |
| **修複** | 在 `_save_current_rule` 中，若 `click_position == 'custom'` 且 `custom_x == 0 and custom_y == 0`，彈出警告要求使用者先選取座標 |
| **驗證** | 選擇 custom 但不選座標 → 儲存 → 看到警告；選取座標後儲存 → 正常 |

### F9: `_heartbeat_loop` 處理 `_restart_ahk` 失敗

| 項目 | 內容 |
|------|------|
| **檔案** | `core/03_ahk_socket.py` |
| **行數** | 76–87 |
| **問題** | `_heartbeat_loop` 呼叫 `_restart_ahk()` 但忽略其回傳值。若重啟失敗，仍僅靠連續 3 次 PING 失敗才觸發緊急停止 |
| **修複** | 若 `_restart_ahk()` 回傳 False，應直接增加 `consecutive_fail` 計數（無需等下一次 PING） |
| **驗證** | 模擬 AHK 崩潰（手動 kill 行程），觀察 `consecutive_fail` 在 restart 失敗後立即增加 |

### F10: AHK 腳本心跳超時自動退出

| 項目 | 內容 |
|------|------|
| **檔案** | `clicker.ahk` |
| **問題** | 若 Python 端意外關閉（非透過 `shutdown()`），AHK 程序會持續存在 |
| **修複** | 在 AHK 端增加心跳超時機制：記錄最後收到指令的時間，若超過 30 秒無任何訊息則自動 `ExitApp`。可在主迴圈底部加入逾時檢查 |
| **驗證** | 啟動後強制結束 Python 行程 → 30 秒後 AHK 應自動退出 |

## Phase 4 — P3 低優先

### F11: 相同 ROI 規則共用 OCR 結果

| 項目 | 內容 |
|------|------|
| **檔案** | `core/05_main_loop.py` |
| **行數** | 128–168 (`_process_rules`) |
| **問題** | 若多條規則使用完全相同的 ROI，每條規則都獨立執行 `recognize`，浪費運算資源 |
| **修複** | 以 ROI tuple `(x, y, w, h)` 為 key 快取 OCR 結果。在 `_process_rules` 迭代規則前，先對不同的 ROI 執行一次 OCR 並暫存結果供後續規則共用。注意全視窗 (`roi=None`) 也視為同一 key |
| **驗證** | 建立兩條相同 ROI 的規則 → 檢查 `_run_engine` 僅被呼叫一次而非兩次 |
